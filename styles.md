# Uncodixify

This document teaches you how to avoid generic AI-made design, without forcing every result into the same structure.

The problem is not only bad UI. The deeper problem is convergence: the same layouts, the same radii, the same dashboard skeletons, the same muted dark palettes, the same “tasteful” card grids, the same fake polish, and the same overused design gestures. A design can obey anti-AI rules and still look repetitive if those rules become a substitute for judgment.

Use this file as a filter, not a template.

Its purpose is to stop obvious AI/Codex defaults, remove lazy aesthetic decisions, and push the work toward something that feels deliberate, human, and appropriate to the actual task. It should never trap the design into one safe visual language.

## Core Principle

Use Uncodixify as:

- a strong list of anti-patterns
- a high-level taste filter
- a cleanup pass against default AI choices

Do **not** use it as:

- a fixed style guide
- a mandatory layout recipe
- a color prison
- a component blueprint
- a reason every design should resemble the last one

The goal is not “make everything look like Linear/Raycast/Stripe/GitHub.”
The goal is: **avoid generic AI design habits, while allowing the result to become whatever it genuinely should become.**

That may mean the output is:

- quiet and minimal
- editorial and expressive
- bold and graphic
- technical and dense
- institutional and sober
- playful and unconventional
- premium and restrained
- experimental and art-directed
- plain and utilitarian

The correct answer depends on the product, audience, medium, brand, and purpose.

## What This Applies To

Apply this guidance to any design-sensitive output, including:

- HTML, CSS, React, Vue, Svelte, and frontend UI
- dashboards, apps, websites, landing pages, and internal tools
- documents, PDFs, slide decks, reports, and visual layouts
- posters, diagrams, presentation systems, and other designed artifacts
- structured text layouts where visual hierarchy matters

This is not only about interface chrome. It also applies to tone, hierarchy, spacing, composition, layout rhythm, and the overall feeling of the output.

## First Principle: Solve the Actual Design Problem

Before choosing visuals, identify:

- what the thing is for
- who it is for
- what tone it should carry
- what medium it lives in
- what constraints already exist
- whether the design should disappear or make an impression
- whether clarity, speed, memorability, trust, warmth, or drama matters most

Every design decision should come from that context first.

If the task calls for restraint, be restrained.
If the task calls for strong art direction, allow strong art direction.
If the task calls for unusual structure, allow unusual structure.
If the task calls for simplicity, do not decorate it.

## The Main Rule

If a choice feels like a default AI-generated move, treat it as suspicious.

Do not ban it automatically just because it is common.
Ban it when it appears because it was easy, generic, decorative, or disconnected from the task.

Choose the cleaner, more intentional option.
Choose the more appropriate option.
Choose the option that makes the work feel authored.

## Anti-Convergence Rule

Do **not** let this document cause all outputs to collapse into the same:

- page structure
- card system
- sidebar width
- spacing rhythm
- font mood
- contrast profile
- border style
- radii scale
- color family
- interaction style
- “tasteful modern” aesthetic

If two different projects solve different problems, they should be allowed to look meaningfully different.

Variation is good when it is earned.

## Creative Freedom Rule

You are allowed to be highly creative.

You are allowed to use:

- asymmetry
- strong editorial hierarchy
- unexpected but coherent compositions
- image-led layouts
- expressive typography
- dense information systems
- brutal simplicity
- cinematic pacing
- playful interaction
- visual quietness
- deliberate ornament
- motion with purpose
- atmosphere and mood

You are **not** allowed to use creativity as an excuse for:

- filler
- trend-chasing
- random style stacking
- decoration without meaning
- structure that hurts usability
- novelty that ignores the brief

Freedom is real, but it must still be justified.

## What To Avoid

These are strong anti-patterns, not automatic replacements with one “safe” answer.

### Generic AI Signals

- soft gradient-heavy “premium” styling used to fake taste
- floating glass shells as the default visual language
- over-rounded everything
- decorative eyebrow labels everywhere
- generic “beautiful SaaS” layouts
- fake seriousness through dark panels and cyan accents
- design decisions made because they are easy to generate
- ornamental copy that explains the interface instead of serving it
- empty visual confidence without product logic
- pseudo-sophisticated spacing that creates dead space
- fake complexity through nested cards and rails
- placeholder charts, stats, or badges that do no real work

### Lazy Structure

- defaulting to a sidebar dashboard when the content does not need it
- defaulting to hero + feature grid + testimonials + CTA because it is familiar
- defaulting to KPI cards as the first move
- defaulting to table + filters + right rail for every tool
- defaulting to centered marketing blocks when the content is operational
- defaulting to one-column mobile stacks with no rethinking of hierarchy
- defaulting to identical cards repeated across the whole page

### Lazy Visual Language

- one radius reused on every element regardless of role
- glow instead of hierarchy
- shadow instead of structure
- blur instead of contrast
- pills everywhere
- blue-black gradients as a shortcut to “modern”
- muted gray-blue text that kills legibility
- decorative status dots, badges, and labels with no informational value
- artificial polish added to weak composition
- over-smoothing everything until it loses character

### Lazy Copy Signals

- startup filler copy
- “operational clarity” type headings
- abstract labels that sound important but say nothing
- little explanatory notes under every section
- faux-product phrases like “live pulse”, “command center”, “night shift”, “control room” unless the product genuinely speaks that way
- decorative small uppercase labels used only to imply hierarchy

## Strong “No” List

These are recurring red flags.

- no glassmorphism as the default answer
- no oversized border radii as the default answer
- no pill-shaped overload
- no fake charts for visual filler
- no hero sections inside internal tools without real purpose
- no decorative dashboard copy
- no dramatic shadows trying to create depth that the layout did not earn
- no random glows, hazes, frosted layers, or gradient borders
- no template-feeling typography
- no cargo-culting “premium” design
- no overpadded layouts used to simulate elegance
- no layout gimmicks that weaken scanning
- no ornamental labels unless they are functionally or brand-wise justified
- no banning of headings entirely, but no decorative heading systems that add fluff instead of information
- no forbidding a color family on principle, but do not drift into predictable AI blue-cyan tech styling unless the brand truly calls for it

## Use Headings Properly

Headings are allowed.

What is **not** allowed is decorative heading behavior that exists only to make the design feel expensive or “designed.”

Avoid:

- tiny eyebrow labels with forced uppercase and letter-spacing
- heading stacks that say little and take too much space
- abstract motivational product copy
- section intros that explain obvious UI

Prefer:

- direct titles
- useful subtitles only when needed
- clear hierarchy
- copy that belongs to the product, not to the generator

## Use Components Deliberately

Do not invent random novelty components for no reason.
Do not blindly clone the same safe component set every time either.

Use familiar, legible component patterns when the task benefits from them.
Depart from them when the design genuinely benefits from a different approach.

Components should feel:

- appropriate
- coherent
- intentional
- tied to the medium
- tied to the content density
- tied to the tone

Not every design needs cards.
Not every design needs a sidebar.
Not every design needs tabs.
Not every design needs badges.
Not every design needs a big top section.

## Layout Guidance

There is no single correct layout.

Choose the layout that best serves the content:

- landing pages may need large sectional pacing
- dashboards may need dense operational clarity
- editorial pages may need typographic hierarchy
- reports may need disciplined grids
- slides may need dramatic composition and restraint
- forms may need calm repetition and trust
- tools may need compact working space
- documents may need quiet structure over interface styling

Do not “invent a new layout” just to look creative.
Do not refuse a new layout when the default pattern is wrong.

Use judgment.

## Normal Is Allowed, Not Required

Simple is good when simple is correct.

A clean, ordinary interface is often the right answer.
But “normal” should not become another trap that makes every result look alike.

Use ordinary building blocks when they help.
Break from them when the problem asks for something else.

The key is this:

- avoid generic AI flourish
- avoid formulaic anti-AI sameness

Both are failures.

## Color Policy

You are not choosing colors to look trendy.
You are choosing colors to support hierarchy, tone, brand, and readability.

Priority order:

1. **Use existing project or brand colors first** if they are provided.
2. If no palette exists, derive a palette from the context, medium, and mood of the task.
3. If needed, use the palettes below as inspiration, not as mandatory defaults.
4. Do not invent chaotic combinations without reason.
5. Do not keep reusing the same familiar palette family just because it feels safe.

Color should support the idea, not replace it.

### Color Rules

- keep contrast strong enough to read comfortably
- let one or two colors do the work, not six
- do not use accent colors everywhere
- avoid “AI modern” gradients unless the concept truly benefits from them
- muted is not automatically tasteful
- bold is not automatically tacky
- neutral is not automatically correct
- calm palettes are useful, but variety is allowed
- the same palette should not appear across unrelated outputs by default

## Typography Policy

Typography should fit the project.

Do not default to the same generic stack every time.
Do not use type choices as a shortcut to “premium.”
Do not mix styles lazily.

Type can be:

- quiet
- technical
- institutional
- editorial
- elegant
- severe
- friendly
- dense
- highly expressive

What matters is that it matches the work.

Avoid typography that feels like it came from a design template rather than a point of view.

## Motion Policy

Motion is allowed.

Do not ban motion.
Ban empty motion.

Use animation when it improves:

- hierarchy
- continuity
- clarity
- delight
- feedback
- narrative pacing

Avoid:

- gratuitous transforms
- bouncy default movement
- ornamental hover theatrics
- motion that tries to compensate for weak structure

Subtle is often good.
Expressive is fine when justified.

## Density Policy

Not every design should breathe the same way.

Some outputs should be airy.
Some should be tight.
Some should be dense and information-rich.
Some should be quiet and sparse.

Do not force one spacing rhythm across every design.
Use density as part of the design language.

## Medium-Specific Guidance

### For UI and Frontend

- keep interfaces readable and purposeful
- avoid fake-product filler
- do not decorate weak information architecture
- use conventional patterns when helpful, not by reflex
- allow more character when the product benefits from it

### For Slides

- do not turn slides into dashboards
- use composition, pacing, contrast, and hierarchy
- allow drama when appropriate
- keep one strong idea per slide when the format benefits from it

### For PDFs and Documents

- think editorially, not like app chrome
- use type, spacing, margins, and rhythm intentionally
- avoid card-ifying everything
- let documents feel like documents when they should

### For Visual Systems and Creative Layouts

- allow stronger direction
- use structure before decoration
- keep the concept coherent
- avoid style soup
- let the work feel authored, not assembled

## Practical Workflow

When designing, do this:

1. understand the real goal
2. identify the medium, audience, and tone
3. decide what kind of visual language actually fits
4. generate or structure the work freely
5. run Uncodixify as a refinement pass
6. remove generic AI defaults
7. check that the result did not collapse into the same familiar anti-AI formula
8. keep only what is clear, appropriate, and alive

## Final Test

Before finalizing, ask:

- does this solve the actual problem?
- does it feel authored rather than generated?
- did I remove generic AI habits?
- did I avoid replacing them with one repetitive house style?
- does the layout fit the content?
- does the palette fit the tone?
- does the typography feel chosen?
- is the motion meaningful?
- is the result distinct enough from previous outputs?
- is the creativity real, or just decorative?

If it still feels like a default AI move, change it.
If it feels like a rigid anti-AI clone of past work, change it too.

That is how you Uncodixify.
